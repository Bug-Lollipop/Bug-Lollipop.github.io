没必要的 currentColor 变量

我们知道 currentColor 变量是个好东西，可以使用当前 color 计算值，IE9 + 都能支持。
实际上， CSS 中很多属性值默认就是 currentColor的表现，我们一般 无需再画蛇添足去再声明这个关键字。如 border 、 text-shadow、 box-shadow 等，尤其是 border。
因此有需要元素 hover 改变颜色和边框的情况，只需要控制 color 的值就可以了，例如：

.btn {
  color: red;
  border: 2px solid;
}

.btn:hover {
  color: blue;
}

而不需要去这样加：
border: 2px solid currentColor

千分位分隔符

千位分隔符 是很常见的需求，但是输入文本千变万化，如何才能准确添加千分符呢？

纯整数情况：

纯整数大概是所有情况里最简单的一种，我们只要正确匹配出千分位就好了。

观察上面的数字，我们可以得出千分位的特征是 到字符串终止位有 3n 个数字，不包括起始位 。于是可以得到这样的函数：

let miliFormat = (num) => {  
    return num && num.toString().replace(/(?=(?!^)(\d{3})+$)/g, ',')
}

但是往往现实没有那么乐观：

小数的情况

遇到小数时，我们的希望只针对整数部分添加千分符，这时问题就变得稍稍有些棘手了。

如果正则引擎支持逆序环视 [2] ，我们可以这样构造正则表达式：

(?<=\d+)(?=(\d{3})+\b)

但是多数语言并不支持逆序环视，所以我们要变通一下：

  1. 拿到小数的整数部分 也就是起始位到小数点（非数字）之间的部分，可以这样实现：

^\d+
2. 为整数部分添加千分符
这一步可以利用我们之前的实现，整合在一起如下：

let miliFormat = (num) => {  
    return num && num
        .toString()
        .replace(/^\d+/, (m) => m.replace(/(?=(?!^)(\d{3})+$)/g, ','))
}

这个函数对整、小数都能正确处理。

CSS writing-mode 的更多使用

writing-mode 通常可以用来实现文字的竖向呈现。
属性对应的关键字有下面几个：

writing-mode: horizonal-tb;  /* 默认值，表示文本流是水平方向的，元素是从上向下的*/
writing-mode: vertical-rl;  /* 表示文本是垂直方向展示 ，阅读顺序是从右向左（right-left）*/
writing-mode: vertival-lr;  /* 表示文本是垂直方向展示 ，阅读顺序是从左向右（left-right）*/

除了竖直排版的时候需要writing-mode，其实还可以实现很多其他的效果。

writing-mode 将页面默认的水平文档流改成了垂直流，基于水平方向的规则都可以适用于垂直方向，比如：

  • margin 合并
  • 用 text-align: center 实现图片垂直居中
  • 使用 text-indent 实现文字下沉效果
  • 可以实现全兼容的 icon-fonts 图标的旋转效果

具体的各个效果可以看这个 http://demo.cssworld.cn/ 中的 （12-2-1） ~ （12-2-5） 里面的代码效果

min-width 与 width 之类的属性之间的相互覆盖

1.超越important

超越 !important 是指 max-width 会覆盖width，并且比 !important 的权重更高。
例如下面的代码：

<img src="1.jpg" style="width: 480px!important">
img {
  max-width: 250px;
}

最终图片呈现的效果是 250px。因为 max-width 覆盖了 width

2.超越最大

超越最大是指 min-width 覆盖max-width，此规则发生在min-width 和max-width 冲突的时候，例如下面： 

.container {
  min-width: 1400px;
  max-width: 1200px;
}

前端实现图片上传

移动端

移动端的H5通过 JsBridge 上传图片，返回base64，然后我们需要将base64 转换成Blob 二进制文件，再发送给后端，代码如下：


export const convertBase64UrlToBlob = dataURI => {
  const arr = dataURI.split(',')
  const mime = arr[0].match(/:(.*?);/)[1]
  
  //去掉base64中的换行符，webkit会自动去除，但是IOS9以及IOS8中不会自动去除，导致转换出错
  const bstr = atob(arr[1].replace(/\s/g, ''))
  let n = bstr.length
  const u8arr = new Uint8Array(n)
  while (n--) {
    u8arr[n] = bstr.charCodeAt(n)
  }
  return new Blob([u8arr], { type: mime })
}

margin 合并

什么是margin 合并

块级元素的上下边距有时会合并成单个边距，这种现象就是 margin 合并。
该现象只发生在和当前文档流方向垂直的方向上。由于默认文档流是水平流，所以发生margin 合并的是垂直方向。

margin 合并的3种场景

相邻元素margin 合并

这个比较常见：

p {
  margin: 1em 0;
}
<p>第一行</p>
<p>第二行</p>

实际效果是 第一行和第二行的间距还是 1em。

【译】如何逃离 async / await 地狱

async/ await 使我们从回调地狱中解脱出来。但是人们又开始被它虐待 - 进入了 async/ await 地狱

在这片文章中，我将会尝试去解释 async/await 地狱是什么，同时分享一些解决方案。

什么是 async/await 地狱

当 Javascript 异步编程的时候时，人们总是使用很多 await 语句，并在函数调用之前总是等待它。这会导致性能问题，因为一个语句多次调用并不依赖前一个语句 - 但是你仍然必须等待前一个语句的完成。

一个 async/ await 地狱的例子

尝试去写一个去实现购买披萨和饮料的功能。这个代码可能是这个样子的：

(async () => {
  const pizzaData = await getPizzaData()    // async call
  const drinkData = await getDrinkData()    // async call
  const chosenPizza = choosePizza()    // sync call
  const chosenDrink = chooseDrink()    // sync call
  await addPizzaToCart(chosenPizza)    // async call
  await addDrinkToCart(chosenDrink)    // async call
  orderItems()    // async call
})()

font-size: 0 与文本隐藏

chrome 中文浏览器有个12px 的字号限制，就是文字的font-size 计算值不能小于12px。

所以设置了小于12px 的字体会被当做12px处理，但是有个例外，那就是0，也就是说，如果font-size: 0 的字号表现就是0，那么文本就可以直接被隐藏掉，并且只能是 font-size: 0，哪怕设置成 fong-size: 0.00001px，都会被当做12px处理。

因此希望隐藏logo对应元素内的文字，除了text-indent 缩进隐藏，还可以用：

.logo {
  font-size: 0;
}
